## O problema que toda aplicação corporativa tem

Bruno transfere R$500 da conta dele pra Murilo. Em SQL, isso vira **dois UPDATEs**:

```sql
UPDATE Conta SET saldo = saldo - 500 WHERE id = 1;  -- conta do Bruno
UPDATE Conta SET saldo = saldo + 500 WHERE id = 2;  -- conta do Murilo
```

**Pergunta**: o que acontece se o **banco crashar** entre os dois UPDATEs?

Resposta sem transação: **Bruno fica R$500 mais pobre**, Murilo não recebe, dinheiro **literalmente some**. Bug catastrófico. Em produção bancária, é demissão e processo.

Com **transação**:

```sql
BEGIN;
UPDATE Conta SET saldo = saldo - 500 WHERE id = 1;
UPDATE Conta SET saldo = saldo + 500 WHERE id = 2;
COMMIT;
```

Agora **ou os dois UPDATEs acontecem juntos, ou nenhum acontece**. Se o banco crashar antes do `COMMIT`, ao reiniciar ele **desfaz** o UPDATE parcial — Bruno volta a ter os R$500. Se crashar depois do `COMMIT`, ambos os UPDATEs **estão garantidos** em disco — Murilo já tem o dinheiro.

Essa garantia é o que torna o modelo relacional **confiável em sistemas reais**. **Toda transação bancária, e-commerce, Pix, ERP** depende disso.

Esta lição cobre:

1. **O que é uma transação** — BEGIN/COMMIT/ROLLBACK
2. **ACID** — Atomicity, Consistency, Isolation, Durability (Härder & Reuter, 1983)
3. **Os 4 Isolation Levels** do ANSI SQL-92
4. **Anomalias de concorrência** — dirty read, phantom, lost update, write skew
5. **MVCC** — como PostgreSQL faz isolamento sem bloquear leitores
6. **Locking** e deadlock detection
7. **Práticas reais** em Spring Boot com `@Transactional`

No fim, você consegue debugar bugs de concorrência (que são os mais sutis do mundo) e configurar isolation level correto pra seu caso.

## A história: como nasceu o conceito

Em **1981**, **Jim Gray** — um dos maiores pesquisadores de bancos de dados de todos os tempos, na época na IBM — publicou *"The Transaction Concept: Virtues and Limitations"* no VLDB em Cannes. Esse paper formalizou o **conceito de transação** como unidade lógica de trabalho com garantias específicas.

Em **dezembro de 1983**, **Theo Härder** e **Andreas Reuter** (da Universidade de Kaiserslautern, Alemanha) publicaram em *ACM Computing Surveys* 15(4), pp. 287-317, o paper **"Principles of Transaction-Oriented Database Recovery"**. **É aí que o acrônimo ACID nasce**:

> *"The four properties, atomicity (A), consistency (C), isolation (I), and durability (D), describe the major highlights of the transaction paradigm, which has influenced many aspects of development in database systems."*

Härder e Reuter **citam Jim Gray como autor do conceito** — eles consolidaram o framework conceitual. **Jim Gray ganhou o Turing Award em 1998** principalmente por esse trabalho.

Curiosidade trágica: **Jim Gray desapareceu em 28 de janeiro de 2007** num passeio de barco solo na Baía de São Francisco, em direção às Farallon Islands. Empresas como Google, Amazon, Microsoft e NASA participaram da busca usando imagens de satélite. **Nunca foi encontrado**. Tinha 63 anos. Em 2012, foi oficialmente declarado morto.

## O que ACID significa

Antes dos isolation levels, entenda as **4 propriedades**:

### A — Atomicity (Atomicidade)

> "Tudo ou nada." A transação inteira é executada com sucesso, **ou nada é aplicado**. Não existe estado parcial.

No exemplo da transferência: ou Bruno perde R$500 **e** Murilo ganha R$500, ou nenhum dos dois acontece. Não pode acontecer só metade.

Implementado via **Write-Ahead Log (WAL)** + rollback. Toda mudança é primeiro logada antes de aplicada na tabela. Se algo dá errado, o log permite **desfazer** (undo) ou **refazer** (redo) operações.

### C — Consistency (Consistência)

> Antes e depois da transação, o banco está num **estado válido** — todas as constraints (PK, FK, CHECK, UNIQUE) são satisfeitas.

Diferente do "C" do CAP theorem (que cobriremos em lição futura), aqui consistência é sobre **regras do banco**, não distribuição.

Se eu tentar inserir um pedido com `cliente_id` que não existe, a FK rejeita e a transação faz rollback. **Nunca o banco fica num estado inconsistente** mesmo que a transação tente.

### I — Isolation (Isolamento)

> Transações **executando em paralelo** não interferem umas nas outras. O resultado é como se tivessem executado **em sequência**.

Esta é a propriedade **mais sutil e mais cara** de implementar — e a que tem **níveis configuráveis** (vamos detalhar). Sem isolamento, transações concorrentes causam bugs absurdos.

### D — Durability (Durabilidade)

> Depois do `COMMIT`, o resultado **persiste mesmo se o servidor crashar**, perder energia, ou falhar de qualquer forma.

Implementado via **fsync** do log e/ou da tabela em disco antes do COMMIT retornar sucesso. Por isso bancos rodam em discos com **fsync confiável** (e por isso SSDs corporativos têm super-capacitor pra garantir escrita em caso de queda de energia).

<Insight>
**Trade-off oculto do ACID**: cada propriedade tem custo de performance. Atomicidade e durabilidade exigem WAL + fsync (escritas em disco). Consistência exige validar constraints. **Isolamento é o mais caro** — pode exigir locks, MVCC, ou re-execução de transações. Por isso bancos **deixam você ajustar isolamento por transação**: paga só o nível necessário. Sistemas NoSQL muitas vezes **abrem mão de algumas dessas propriedades** (eventual consistency, ausência de ACID multi-document) pra ganhar performance/escala.
</Insight>

## Comandos básicos: BEGIN, COMMIT, ROLLBACK

Toda transação tem 3 comandos:

```sql
BEGIN;              -- inicia transação (também: START TRANSACTION)
-- ... operações ...
COMMIT;             -- confirma todas as mudanças
-- OU
ROLLBACK;           -- desfaz tudo desde o BEGIN
```

Exemplo completo:

```sql
BEGIN;

UPDATE Conta SET saldo = saldo - 500 WHERE id = 1;

-- Verificar se saldo ficou negativo
SELECT saldo FROM Conta WHERE id = 1;
-- Se for negativo: ROLLBACK; sair sem fazer transferência

UPDATE Conta SET saldo = saldo + 500 WHERE id = 2;

COMMIT;
```

**Savepoints** permitem rollback parcial:

```sql
BEGIN;
UPDATE Pedido SET status = 'PAGO' WHERE id = 101;
SAVEPOINT antes_envio;

UPDATE Pedido SET status_envio = 'EM_TRANSITO' WHERE id = 101;
-- Algo deu errado no envio
ROLLBACK TO SAVEPOINT antes_envio;
-- Pedido continua PAGO, mas status_envio volta ao original

COMMIT;
```

**Em Spring Boot**, isso geralmente é abstraído com `@Transactional`:

```java
@Service
public class TransferenciaService {

    @Transactional
    public void transferir(Long origemId, Long destinoId, BigDecimal valor) {
        Conta origem = contaRepo.findById(origemId).orElseThrow();
        Conta destino = contaRepo.findById(destinoId).orElseThrow();

        origem.debitar(valor);  // pode lançar exceção se saldo insuficiente
        destino.creditar(valor);

        contaRepo.save(origem);
        contaRepo.save(destino);
        // commit automático ao retornar sem exceção
        // rollback automático se qualquer exceção for lançada
    }
}
```

Sob o capô, Spring abre transação no início do método e fecha no fim. **Exceção lançada = rollback automático**.

## Concorrência: o problema que isolation resolve

Considere 2 usuários transferindo dinheiro **da mesma conta simultaneamente**:

```
Bruno (T1)                          Murilo (T2)
============                        ============
saldo = 1000

BEGIN                               BEGIN
saldo = SELECT(conta=X) → 1000
                                    saldo = SELECT(conta=X) → 1000
                                    saldo = 1000 - 300 = 700
                                    UPDATE: saldo = 700
                                    COMMIT
saldo = 1000 - 500 = 500
UPDATE: saldo = 500
COMMIT

-- saldo final = 500
-- MAS saiu (-300) + (-500) = -800 ESPERADO 200
```

Bug clássico: **lost update**. O resultado correto seria saldo = 200, mas terminou em 500 porque T1 leu o saldo **antes** de T2 atualizar — ignorou completamente o débito de Murilo.

Isso é o que isolation levels **previnem** (em diferentes graus).

## Os 4 Isolation Levels do ANSI SQL-92

ANSI definiu 4 níveis, do mais "fraco" pro mais "forte":

| Isolation Level | Dirty Read | Non-repeatable Read | Phantom Read |
|---|---|---|---|
| **READ UNCOMMITTED** | Pode | Pode | Pode |
| **READ COMMITTED** | Não | Pode | Pode |
| **REPEATABLE READ** | Não | Não | Pode |
| **SERIALIZABLE** | Não | Não | Não |

Antes de explicar cada nível, vamos entender as 3 anomalias:

### Anomalia 1: Dirty Read

> Uma transação lê dados que outra transação **ainda não commitou**. Se essa outra fizer rollback, sua leitura é "suja" (baseada em dado que nunca existiu).

```
T1: BEGIN; UPDATE Conta SET saldo = 100 WHERE id = 1;
T2: BEGIN; SELECT saldo FROM Conta WHERE id = 1;   -- lê 100 (não commitado)
T1: ROLLBACK;                                       -- saldo volta a 1000
T2: usa o valor 100 que nunca existiu de verdade
```

### Anomalia 2: Non-repeatable Read (Fuzzy Read)

> Uma transação lê o mesmo dado **duas vezes** e obtém valores diferentes, porque outra transação commitou no meio.

```
T1: BEGIN;
T1: SELECT saldo FROM Conta WHERE id = 1;          -- lê 1000
T2: BEGIN; UPDATE Conta SET saldo = 500 WHERE id = 1; COMMIT;
T1: SELECT saldo FROM Conta WHERE id = 1;          -- lê 500 (mudou!)
```

### Anomalia 3: Phantom Read

> Uma transação lê um **conjunto de linhas** com filtro, e na segunda leitura aparecem linhas novas (fantasmas) que satisfazem o filtro, porque outra transação inseriu.

```
T1: BEGIN;
T1: SELECT COUNT(*) FROM Pedido WHERE valor > 1000;   -- retorna 5
T2: BEGIN; INSERT INTO Pedido(valor) VALUES (1500); COMMIT;
T1: SELECT COUNT(*) FROM Pedido WHERE valor > 1000;   -- retorna 6 (fantasma!)
```

### READ UNCOMMITTED — sem isolamento

Permite **todas** as anomalias. Praticamente ninguém usa em produção. **PostgreSQL não implementa de verdade** — trata como READ COMMITTED.

### READ COMMITTED — o default da maioria

Cada `SELECT` vê apenas dados **commitados**. **Default do PostgreSQL e Oracle**. Previne dirty read, mas permite non-repeatable read e phantom.

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

Caso de uso: queries OLTP típicas, onde tolerar variações entre selects da mesma transação é OK.

### REPEATABLE READ — leitura consistente

Toda leitura dentro da transação vê o **snapshot do banco no início da transação**. Previne dirty read e non-repeatable read. **Default do MySQL/InnoDB**.

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

**Detalhe importante sobre PostgreSQL**: o "REPEATABLE READ" do PostgreSQL na verdade implementa **Snapshot Isolation** (mais forte que o REPEATABLE READ definido na ANSI) — previne phantom também. Veremos snapshot isolation abaixo.

### SERIALIZABLE — isolamento total

Garante que o resultado é **como se as transações tivessem executado em sequência**. Previne todas as anomalias. **Mais caro** em performance — pode causar mais aborts e re-execuções.

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

Caso de uso: operações críticas com consistência absoluta — transferências financeiras complexas, contagem de estoque, lock crítico.

<Aviso>
**Trade-off real**: SERIALIZABLE é a opção **mais segura** mas a **mais cara**. Em PostgreSQL, usa Serializable Snapshot Isolation (SSI) que pode abortar transações em conflito — sua aplicação precisa estar preparada pra **fazer retry**. SQL Server tradicional usa locking em serializable, podendo causar lock waits longos. Use SERIALIZABLE com **intenção explícita** em operações críticas, não como default global.
</Aviso>

## Anomalias além da ANSI: Lost Update e Write Skew

O paper **Berenson et al. (1995)** de Microsoft Research e Jim Gray, *"A Critique of ANSI SQL Isolation Levels"* (SIGMOD '95), mostrou que os 3 fenômenos da ANSI **não cobrem todas as anomalias**. Os autores: Hal Berenson, Phil Bernstein (Microsoft), Jim Gray (que tinha ido pra UC Berkeley), Jim Melton (Sybase), Elizabeth & Patrick O'Neil (UMass).

### P0 — Dirty Write (escrita suja)

> Duas transações escrevem na mesma linha sem que a primeira tenha commitado. ANSI não menciona, mas é a **mais básica** das anomalias — deve ser prevenida em **todo** isolation level (inclusive READ UNCOMMITTED).

### Lost Update — o bug clássico das transferências

Já vimos no exemplo da transferência. Em isolation levels fracos, T1 lê valor, T2 lê valor, T1 escreve novo valor, T2 escreve novo valor — **a escrita de T1 é perdida**.

**Soluções**:

1. **SELECT FOR UPDATE** — bloqueia a linha pra outras transações até o COMMIT
2. **Optimistic locking** com versão — coluna `version` que incrementa, UPDATE só funciona se versão bater
3. **UPDATE atômico** — em vez de `SELECT + UPDATE`, fazer `UPDATE saldo = saldo - 500` direto (banco resolve atomicamente)

```sql
-- 1. Pessimistic locking
BEGIN;
SELECT * FROM Conta WHERE id = 1 FOR UPDATE;  -- bloqueia linha
UPDATE Conta SET saldo = saldo - 500 WHERE id = 1;
COMMIT;
```

Em Spring Boot/JPA: `@Lock(LockModeType.PESSIMISTIC_WRITE)`.

```sql
-- 2. Optimistic locking
UPDATE Conta SET saldo = ?, versao = versao + 1
WHERE id = 1 AND versao = ?;  -- se outro update mudou versao, UPDATE não afeta linha
-- aplicação verifica rows affected; se 0, faz retry
```

Em JPA: `@Version` em campo Long da entity.

### Write Skew — anomalia do Snapshot Isolation

Caso famoso: regra do hospital "sempre pelo menos 1 médico de plantão". 2 médicos online: Bruno e Murilo. Cada um simultaneamente pede "tirar plantão":

```
T1 (Bruno):                          T2 (Murilo):
SELECT count(*) FROM Plantao         SELECT count(*) FROM Plantao
WHERE ativo = true;                  WHERE ativo = true;
-- retorna 2                         -- retorna 2

UPDATE Plantao SET ativo = false     UPDATE Plantao SET ativo = false
WHERE medico_id = 'bruno';           WHERE medico_id = 'murilo';

COMMIT                               COMMIT

-- Resultado: 0 médicos no plantão!
```

**Snapshot Isolation permite isso** — cada transação vê o snapshot inicial (2 médicos), nenhuma modifica a mesma linha. **Apenas SERIALIZABLE previne** (detecta o conflito de "leituras que afetam a decisão de escrita").

Esse é o exemplo mais famoso do paper de Berenson e demonstra por que SI ≠ Serializable.

## MVCC: como PostgreSQL faz isolamento sem bloquear leitores

**MVCC** = **Multi-Version Concurrency Control**. Conceito proposto na **tese de PhD de David P. Reed no MIT em 1978**. PostgreSQL, Oracle, SQL Server (read committed snapshot), MySQL/InnoDB todos usam variantes hoje.

### A ideia central

Em vez de bloquear leitores quando alguém escreve, o banco **mantém múltiplas versões** de cada linha. Cada transação vê a **versão apropriada** baseada em quando começou.

**Resultado**: **leitores nunca bloqueiam escritores, e escritores nunca bloqueiam leitores**. Cada um vê seu snapshot consistente.

### Como funciona no PostgreSQL

Cada linha tem dois campos invisíveis:
- **`xmin`**: ID da transação que criou esta versão
- **`xmax`**: ID da transação que deletou/atualizou esta versão (ou 0)

Quando você faz `UPDATE`, o PostgreSQL **não modifica a linha existente** — ele:
1. Marca a linha antiga com `xmax = sua_transacao_id`
2. Cria uma **nova linha** com `xmin = sua_transacao_id`

### Consequências práticas no PostgreSQL

1. **`UPDATE` é fisicamente um INSERT + marcação da linha antiga como dead**
2. **Tabelas crescem com UPDATEs** mesmo sem novos dados — bloat
3. **`VACUUM`** é o processo de garbage collection — remove linhas "dead" (sem transação ativa que ainda precise delas). Roda automaticamente (autovacuum)
4. **Leitores nunca bloqueiam escritores e vice-versa** — diferente de SQL Server tradicional onde locks de leitura são comuns
5. **Snapshot Isolation real** — o que o PostgreSQL chama de REPEATABLE READ

### Bloat e VACUUM

Em tabelas com muito UPDATE, o tamanho físico cresce continuamente. Se autovacuum não acompanha, tabelas viram **bloated** — espaço alocado em disco muito maior que dados úteis.

```sql
-- Verificar bloat
SELECT relname, n_live_tup, n_dead_tup, n_dead_tup / NULLIF(n_live_tup::float, 0) AS dead_ratio
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

-- Forçar VACUUM
VACUUM ANALYZE Conta;

-- VACUUM FULL — reescreve a tabela inteira (BLOQUEIA — não usar em prod sem cuidado)
VACUUM FULL Conta;

-- Alternativa sem lock: pg_repack (extensão), tools como pg_squeeze
```

## Locking: 2PL e Deadlock

Em isolation levels mais altos (especialmente SERIALIZABLE com locking, como SQL Server tradicional), o banco usa **locks** pra garantir isolamento.

### Two-Phase Locking (2PL)

Inventado em **K. P. Eswaran, J. N. Gray, R. A. Lorie, I. L. Traiger (1976)** — paper *"The Notions of Consistency and Predicate Locks in a Database System"* (CACM 19(11)).

Toda transação tem **duas fases**:
1. **Growing phase**: adquire locks, **nunca libera**
2. **Shrinking phase**: libera locks, **nunca adquire mais**

Tipos de lock:
- **Shared (S)** — lock de leitura. Múltiplos S simultâneos OK
- **Exclusive (X)** — lock de escrita. Bloqueia tudo

Matriz de compatibilidade:

|  | S existente | X existente |
|---|---|---|
| Pede S | OK | Espera |
| Pede X | Espera | Espera |

### Deadlock

Bug clássico: 2 transações esperando uma pela outra.

```
T1: lock conta 1, depois lock conta 2
T2: lock conta 2, depois lock conta 1

T1 lockou conta 1, espera conta 2
T2 lockou conta 2, espera conta 1
=> deadlock infinito
```

Bancos detectam isso automaticamente (graph de espera) e **escolhem uma vítima** pra abortar — sua aplicação pega exceção e precisa fazer retry.

**Prevenção**: sempre adquirir locks na **mesma ordem** (ex: ordenar por ID antes de acessar). Em código:

```java
@Transactional
public void transferir(Long origemId, Long destinoId, BigDecimal valor) {
    // Garante ordem fixa — evita deadlock
    Long primeiro = Math.min(origemId, destinoId);
    Long segundo = Math.max(origemId, destinoId);

    Conta a = contaRepo.findByIdForUpdate(primeiro);
    Conta b = contaRepo.findByIdForUpdate(segundo);

    if (origemId < destinoId) {
        a.debitar(valor); b.creditar(valor);
    } else {
        b.debitar(valor); a.creditar(valor);
    }
}
```

## Spring Boot: `@Transactional` na prática

Em Spring Boot, transações são abstraídas pela anotação `@Transactional`:

```java
@Service
public class PedidoService {

    // Default: propagation=REQUIRED, isolation=DEFAULT (do banco), rollback em RuntimeException
    @Transactional
    public Pedido criar(PedidoDTO dto) {
        Pedido p = new Pedido(dto);
        pedidoRepo.save(p);
        estoqueService.reservar(dto.getItens());  // outra transação? mesma?
        return p;
    }

    // Isolation explícito
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void operacaoCritica() { ... }

    // Rollback em exceção checada também
    @Transactional(rollbackFor = Exception.class)
    public void operacaoComCheckedException() throws IOException { ... }

    // Apenas leitura — otimização
    @Transactional(readOnly = true)
    public Pedido buscar(Long id) {
        return pedidoRepo.findById(id).orElseThrow();
    }
}
```

### Propagação

`propagation` define o que acontece se já existe uma transação ativa:

| Valor | Comportamento |
|---|---|
| `REQUIRED` (default) | Usa a existente, ou cria nova se não houver |
| `REQUIRES_NEW` | **Sempre cria nova**, suspende a anterior |
| `NESTED` | Cria savepoint dentro da existente |
| `SUPPORTS` | Usa se existir, sem transação se não houver |
| `NEVER` | Falha se houver transação ativa |

**Bug clássico**: usar `REQUIRES_NEW` pra logging dentro de uma transação que faz rollback — o log persiste mesmo que a transação principal aborte. Útil pra auditoria.

### Self-invocation: a pegadinha clássica

`@Transactional` funciona via **proxy**. Quando você chama um método **`@Transactional` a partir de outro método na mesma classe**, o proxy **é ignorado** — a transação não é aberta.

```java
@Service
public class PedidoService {

    public void processar(Long id) {
        // Esse @Transactional NÃO é aplicado!
        salvarComTransacao(id);
    }

    @Transactional
    public void salvarComTransacao(Long id) {
        pedidoRepo.save(new Pedido(id));
    }
}
```

**Solução**: injetar a própria classe, ou separar em outro service.

## Recap

**Transações** garantem consistência em operações que precisam ser **atômicas**. Comandos: `BEGIN`, `COMMIT`, `ROLLBACK`, `SAVEPOINT`. Em Spring Boot, abstraídas com `@Transactional`.

**ACID** (cunhado por **Theo Härder e Andreas Reuter** em **dezembro de 1983**, paper *"Principles of Transaction-Oriented Database Recovery"*, ACM Computing Surveys 15(4)):

- **Atomicity** — tudo ou nada (implementado via WAL + rollback)
- **Consistency** — banco sempre em estado válido (constraints garantidas)
- **Isolation** — transações concorrentes não interferem (níveis configuráveis)
- **Durability** — após COMMIT, dado persiste mesmo com crash (fsync)

**Jim Gray** desenvolveu o conceito original em **1981** (paper VLDB Cannes), recebeu Turing Award em 1998. Desapareceu em 2007 num passeio de barco solo na Baía de São Francisco; nunca foi encontrado.

**4 Isolation Levels** do ANSI SQL-92:

| Nível | Dirty | Non-rep | Phantom |
|---|---|---|---|
| READ UNCOMMITTED | ✗ permite | ✗ permite | ✗ permite |
| READ COMMITTED | ✓ previne | ✗ permite | ✗ permite |
| REPEATABLE READ | ✓ previne | ✓ previne | ✗ permite |
| SERIALIZABLE | ✓ previne | ✓ previne | ✓ previne |

PostgreSQL e Oracle default = READ COMMITTED. MySQL/InnoDB default = REPEATABLE READ. PostgreSQL "REPEATABLE READ" na verdade é **Snapshot Isolation** (mais forte que ANSI).

**Berenson et al. (1995)** mostrou que ANSI **não cobre** Lost Update e Write Skew. Snapshot Isolation previne phantom mas **permite write skew** — caso famoso dos médicos de plantão. Apenas SERIALIZABLE previne tudo.

**MVCC** — proposto por David P. Reed na tese de PhD do MIT em 1978. PostgreSQL mantém múltiplas versões via `xmin`/`xmax`. Leitores nunca bloqueiam escritores. UPDATEs criam novas linhas; VACUUM limpa as antigas. Pode causar **bloat** se autovacuum não acompanhar.

**Locking 2PL** (Eswaran, Gray, Lorie, Traiger 1976) — duas fases: growing (adquire) e shrinking (libera). Locks shared (leitura) e exclusive (escrita). **Deadlock**: bancos detectam graph de espera, abortam vítima, aplicação faz retry. Prevenção: sempre adquirir locks na mesma ordem.

**Spring Boot**: `@Transactional` com `isolation`, `propagation`, `rollbackFor`, `readOnly`. **Pegadinha self-invocation**: chamada de método na mesma classe ignora o proxy → transação não abre.
